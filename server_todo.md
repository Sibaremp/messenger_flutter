# Backend TODO — Caspian College Messenger

Этот файл описывает всё, что нужно реализовать / исправить на сервере,
чтобы клиент (Flutter) работал корректно. Приоритеты расставлены.

## 🔴 КРИТИЧНО — без этого фичи сломаны

### 1. Добавить `userId` в список участников чата

**Затронутые эндпоинты:**
- `GET /chats` (список чатов)
- `GET /chats/{id}` (один чат)
- SignalR событие `chat_updated`

**Проблема:** Клиент получает `members[]` без поля `userId`. Из-за этого
@упоминания отправляются с `userId = displayName` вместо UUID → сервер
отвергает с 400 «User X is not a member of this chat».

**Что сделать:** Добавить поле `userId` (UUID пользователя) в каждый
объект участника в ответе.

```json
// Сейчас (неправильно):
"members": [
  { "name": "Иванов Иван", "role": "member", "group": "CS-101" }
]

// Должно быть:
"members": [
  {
    "userId": "3fa85f64-5717-4562-b3fc-2c963f66afa6",
    "name": "Иванов Иван",
    "role": "member",
    "group": "CS-101"
  }
]
```

### 2. Добавить `unreadCount` в ответ со списком чатов

**Затронутые эндпоинты:**
- `GET /chats`

**Проблема:** Клиент показывает бейдж непрочитанных сообщений —
поле `unreadCount` в модели `Chat` есть, но сервер его не присылает,
поэтому бейдж всегда 0.

**Что сделать:** Для каждого чата считать количество сообщений,
отправленных после последнего прочтения текущим пользователем,
и возвращать как `unreadCount`.

```json
{
  "id": "chat-uuid",
  "name": "Группа CS-101",
  "unreadCount": 5,
  ...
}
```

**Сопутствующее:** Нужен механизм «прочитал до этого сообщения».
Варианты:
- `POST /chats/{id}/read` — пометить чат как прочитанный (вызывается
  при открытии чата)
- Или хранить `lastReadAt` / `lastReadMessageId` на пользователя

Клиент вызывает `_loadChats()` после закрытия экрана чата —
`unreadCount` должен обнулиться к этому моменту.

---

### 3. Корректная валидация @упоминаний в сообщениях

**Затронутый эндпоинт:** `POST /chats/{id}/messages`

**Проблема:** Сервер проверяет `userId` из массива `mentions` и падает
с 400, если это не UUID, или находит пользователя по имени → пингует
«призрака».

**Что сделать:**

1. Принимать `mentions` как опциональное поле — если массив пустой
   или отсутствует, не ругаться.

2. Валидировать `userId` из `mentions` строго по UUID и по членству
   в чате. Если userId не найден — **игнорировать** это упоминание
   (не падать с 400, просто не слать уведомление).

3. Специальный случай `userId == "all"` — отправить уведомление
   всем участникам чата.

**Тело запроса от клиента:**
```json
{
  "text": "Привет @Иванов, посмотри задание",
  "mentions": [
    {
      "userId": "3fa85f64-5717-4562-b3fc-2c963f66afa6",
      "username": "Иванов Иван",
      "offset": 7,
      "length": 13
    }
  ]
}
```

`offset` и `length` — позиция токена `@username` в тексте (включая `@`).
Можно хранить для подсветки на других клиентах.

## 🟡 ВАЖНО — уведомления

### 4. SignalR: уведомлять упомянутого пользователя

**Когда:** После успешного сохранения сообщения с непустым `mentions[]`.

**Что сделать:** Для каждого `userId` из `mentions` найти активные
SignalR-соединения этого пользователя и отправить событие.

Можно переиспользовать существующее `message_received` — клиент сам
смотрит на `message.mentions` и добавляет уведомление. Либо отдельный
тип события (чище):

```json
// Вариант А: переиспользовать message_received (уже работает на клиенте)
{ "type": "message_received", "chatId": "...", "message": { ... } }

// Вариант Б: отдельное событие (опционально)
{ "type": "message_mention", "chatId": "...", "message": { ... } }
```

Для `userId == "all"` — разослать всем участникам чата.

---

### 5. SignalR: уведомлять при ответе на сообщение

**Когда:** Сообщение содержит `replyTo`, и `replyTo.senderName`
совпадает с именем другого пользователя.

**Что сделать:** Найти оригинальное сообщение по `replyTo.messageId`,
определить его автора, отправить ему `message_received` (или отдельный
`message_reply`).

Клиент уже обрабатывает это: при получении `MessageReceived` проверяет
`message.replyTo?.senderName == currentUser.name` → добавляет
уведомление типа «ответ».

## 🟢 ЖЕЛАТЕЛЬНО — управление устройствами (сессии)

### 6. REST API для сессий

| Метод | Путь | Описание |
|-------|------|----------|
| `GET` | `/auth/sessions` | Список активных сессий пользователя |
| `DELETE` | `/auth/sessions/{sessionId}` | Завершить конкретную сессию |
| `DELETE` | `/auth/sessions` | Завершить все сессии (включая текущую) |

**Формат ответа `GET /auth/sessions`:**
```json
[
  {
    "sessionId": "sess-uuid",
    "deviceName": "iPhone 14 Pro",
    "platform": "iOS",
    "location": "Баку, Азербайджан",
    "lastActivity": "2026-04-20T10:30:00Z",
    "isCurrent": true
  },
  {
    "sessionId": "sess-uuid-2",
    "deviceName": "Chrome · Windows 11",
    "platform": "Web",
    "location": "Сумгайыт, Азербайджан",
    "lastActivity": "2026-04-19T18:00:00Z",
    "isCurrent": false
  }
]
```

**`isCurrent: true`** — сессия, чей JWT-токен используется в текущем
запросе. Клиент показывает её первой с бейджем «Это устройство».

---

### 7. SignalR: событие `session_terminated`

**Когда:**
- Администратор или другой клиент завершил сессию через
  `DELETE /auth/sessions/{id}`
- Истёк 30-дневный период неактивности

**Формат события:**
```json
{ "sessionId": "sess-uuid", "isCurrent": false }
```

Если `isCurrent: true` — клиент разлогинивается и показывает экран
входа.

---

### 8. Авто-истечение сессий (cron / background job)

**Задача:** Раз в сутки удалять сессии, у которых `lastActivity`
старше 30 дней. При удалении — отправлять SignalR-событие
`session_terminated` затронутым клиентам (если они ещё онлайн).

---

### 9. Push-уведомление при входе с нового устройства

**Когда:** Успешная аутентификация с устройства/браузера, который
ранее не использовался (новая сессия).

**Что делать:** Отправить push-уведомление (FCM/APNs) на все
остальные активные устройства пользователя:

```
📱 Новый вход
Вход выполнен с устройства "Chrome · Windows 11"
Баку, Азербайджан · только что
```

Если пользователь не ожидал этого входа — он может перейти в
«Управление устройствами» и завершить подозрительную сессию.

## Контекст клиентской стороны

- Клиент парсит `userId` из поля `userId` **или** `id` в объекте участника
- Клиент **не отправляет** mentions в личных чатах (только в группах/сообществах)
- Клиент **не отправляет** mention если `userId == username` (нет реального UUID) —
  это защита от призрачных пингов
- `unreadCount = 0` после вызова `POST /chats/{id}/read` (клиент делает это
  при открытии чата)
- SignalR события должны содержать поле `"type"` как первый ключ —
  клиент переключается по нему
